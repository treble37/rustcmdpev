# ADR-0001: v0.2.0 supersedes the legacy 0.1.0 release

- Status: accepted
- Date: 2026-07-05
- Deciders: Bruce Park (treble37)

## Context

`rustcmdpev 0.1.0` was published to crates.io on 2020-10-06 from the original
port of [gocmdpev](https://github.com/simon-engledew/gocmdpev). The codebase has
since been rewritten as a two-crate workspace (`rustcmdpev` CLI +
`rustcmdpev-core` library), and the workspace manifests carried `0.1.0` during
that rewrite.

crates.io versions are immutable: a version number, once published, can never be
reused or replaced — even if yanked. The rewrite therefore cannot ship as
`0.1.0`, and any doc that implied it would was wrong.

`rustcmdpev-core` has never been published; it gets its first crates.io release
alongside this one.

## Decision

- The rewrite ships as **v0.2.0** for both crates.
- `rustcmdpev-core` debuts on crates.io at `0.2.0` (skipping `0.1.0`) so the two
  workspace crates stay version-aligned.
- The crates.io listing for `rustcmdpev 0.1.0` remains live as the legacy 2020
  release; it is not yanked.
- Normal pre-1.0 semver rules apply from `0.2.0` onward (see
  [Versioning and Tags](versioning.md)): minor bumps may break, patch bumps do
  not.
- Related decision (D7, 2026-07-05): the project spec, task breakdown, and
  requirements/todos planning docs live in the maintainer's project vault, not
  in this repository — deliberately, not by drift. The repo's `docs/` is
  user-facing documentation only; planning artifacts surface here as ADRs and
  release reports.

## Consequences

- No document may claim the rewrite ships as `0.1.0`; references to `0.1.0` in
  this repo must describe it as the legacy 2020 release.
- Downstream users upgrading from the legacy `0.1.0` should expect breaking
  changes across the `0.1 → 0.2` minor bump, which is exactly what pre-1.0
  semver signals.
- The internal dependency pin is
  `rustcmdpev-core = { path = "../rustcmdpev-core", version = "0.2.0" }`; both
  manifests move together on every release.
